1. Corrección de bugs existentes

1.1 Import Excel duplicaba alumnos Resuelto

Problema: Al reimportar un Excel, se creaban registros duplicados en la tabla Alumnos porque VerificaDuplicado() solo verificaba en REGISTRO_ALUMNOS (inscripción al curso) pero no en Alumnos (tabla base).

Solución: Se agregó AlumnosDAL.GetByOtecIdAndRut() que busca el alumno por RUT + OtecId. El import ahora reutiliza alumnos existentes y solo crea nuevos cuando no existen.

Archivos modificados

DAL/AlumnosDAL.cs — Nuevo método GetByOtecIdAndRut() Services/AlumnosService.cs — CommitImportAsync ahora busca antes de crear Controllers/AlumnosController.cs — BulkImport con misma lógica

1.2 UpdateRegistro perdía observaciones Resuelto

Problema: UpdateRegistro borraba y recreaba el REGISTRO_ALUMNOS sin pasar las observaciones, perdiendo los datos de variables del certificado.

Solución: Se reemplazó delete+insert por un UPDATE que preserva los datos.

Archivos modificados

DAL/REGISTRO_ALUMNOS.cs — Nuevo método Update() Controllers/AlumnosController.cs — UpdateRegistro usa Update en vez de delete+insert DTOs/19SharedDTOs.cs — Agregado campo observaciones a UpdateAlumnoRegistroRequest

1.3 SQL roto en CertificadoDAL.update Resuelto

Problema: El SQL tenía un paréntesis sobrante (UPDATE Certificados() y usaba coma en vez de AND en el WHERE.

Archivos modificados

DAL/CertificadoDAL.cs — SQL corregido

2. Eliminación del bloqueo de layout

Resuelto

Problema: IsLayoutBloqueado impedía modificar cursos después de la primera emisión de certificados. Los usuarios se veían obligados a recrear cursos y resubir alumnos para hacer cambios simples como corregir un acento.

Solución: Se eliminó toda la lógica de bloqueo. El layout y todos los datos del curso son siempre editables. Los certificados ya emitidos quedan inmutables con los datos del momento de emisión.

Regla de negocio establecida

Modificar un curso o alumno solo afecta certificados nuevos o re-emitidos. Los certificados ya emitidos nunca se alteran retroactivamente.

Archivos modificados

Services/LayoutService.cs — Eliminado chequeo de IsLayoutBloqueado Services/CursosService.cs — LayoutJson siempre actualizable Services/UpdateCursoRequest.cs — Agregado campo LayoutJson

3. Sistema de cuotas comerciales

Implementado

Sistema para limitar la emisión de certificados por empresa, controlado por el administrador del sitio.

Modelo de datos

TablaPropósito
OtecQuotaConfigToggle on/off de cuota por empresa
OrdenCompraCantidad + fecha vencimiento. Solo una activa por empresa
OrdenCompraHistorialAuditoría: CREATED, SUPERSEDED, QUOTA_CONSUMED

Reglas de negocio

  • Cuota OFF = emisión libre (ilimitada)
  • Cuota ON = solo emite con saldo disponible y orden vigente
  • Nueva orden reemplaza automáticamente la anterior (va al historial)
  • Órdenes no se editan ni cancelan
  • Emisiones nuevas y re-emisiones consumen cuota
  • Vence por cantidad agotada O fecha expirada (lo que ocurra primero)

5. Validaciones de plantillas

Implementado

Variables obligatorias

Al subir un .docx como plantilla, se valida que contenga (case sensitive):

  • {{NOMBRE}} — fijada en tarjeta Alumno
  • {{RUT}} — fijada en tarjeta Alumno
  • {{NOMBRE_CURSO}} — fijada en tarjeta Curso
  • {{QR}} — fijada en tarjeta Curso

Si falta alguna, se corta el flujo y se indica cuál falta.

Edición de plantilla

Al reemplazar el .docx de una plantilla existente:

  • Se permiten cambios de formato y textos estáticos
  • No se permite agregar, eliminar ni renombrar variables
  • Se valida que el conjunto de variables sea idéntico al original

Clasificación drag & drop

Las variables obligatorias aparecen con candado y no se pueden mover. El resto se clasifica libremente.

Archivos modificados

Frontend: PlantillaWizard.tsx — Reescrito completamente

6. Reorganización del wizard de cursos

Implementado

Cambio principal: Se eliminó el paso "Cargar Alumnos" del wizard de cursos. Los alumnos se gestionan exclusivamente desde la sección Alumnos del curso.

Justificación: Tener la carga de alumnos en dos lugares (wizard de curso + pantalla de alumnos) generaba confusión e inconsistencias. Además, obligar a tener el Excel listo antes de crear el curso no refleja el flujo real de trabajo.

Archivos modificados

Frontend: CursoWizard.tsx — Eliminado paso y lógica de import

9. Infraestructura y deploy

Estructura del servidor confirmada

/root/qrcerts/
  Dockerfile          ← Multi-stage (pero producción usa publish/)
  docker-compose.yml  ← sqlserver + qrcerts-app
  publish/            ← Backend compilado (.dll)
    wwwroot/          ← Archivos estáticos
      app/            ← Frontend React (dist)
  uploads/            ← Volume montado (plantillas .docx + imágenes)

Flujo de deploy correcto

  1. Local: dotnet publish -c Release -o ../publish
  2. FileZilla: subir publish/QRCerts.Api.dll + .pdb a /root/qrcerts/publish/
  3. FileZilla: subir frontend dist/ a /root/qrcerts/publish/wwwroot/app/
  4. FileZilla: subir estáticos a /root/qrcerts/publish/wwwroot/
  5. SSH: cd /root/qrcerts && docker compose up -d --build --no-deps qrcerts-app
  6. No pisar appsettings.json de producción

Conexión a BD desde el container

El DALBase lee de appsettings.json (no de variables de entorno). La connection string de producción es:

Server=sqlserver,1433;Database=QRCerts;User Id=sa;Password=QRCerts2024!Strong;TrustServerCertificate=True
